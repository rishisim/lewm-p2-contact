## PushT Dataset Structure

Important caveat: I did not download the 13 GB compressed PushT dataset, so this is inferred from the LeWM configs plus the `stable-worldmodel` collection/reader/writer/environment code.

LeWM training config for PushT loads:

- `pixels`
- `action`
- `proprio`
- `state`

LeWM eval for PushT caches/uses:

- `action`
- `proprio`
- `state`
- plus goal fields generated by slicing future rows, notably `goal_state`

In `stable-worldmodel`, `World.collect()` records per-step values from the env `info` dict. The wrapper also adds bookkeeping and rendered images. For PushT, the environment exposes:

- `pixels`: rendered RGB frame, resized to 224x224 by `MegaWrapper`; HDF5 stores image arrays, Lance stores JPEG blobs.
- `action`: raw 2D action per env step on disk. Dataset loading with `frameskip=5` reshapes this into action blocks, so each model time step sees 5 actions x 2 dims = 10 dims for PushT.
- `proprio`: shape `(4,)`, agent position `(x, y)` and agent velocity `(vx, vy)`.
- `state`: shape `(7,)`, `[agent_x, agent_y, block_x, block_y, block_angle, agent_vx, agent_vy]`.
- `goal`: goal image.
- `goal_state`: goal state vector, same 7D format.
- `goal_proprio`: goal proprio vector.
- `pos_agent`, `vel_agent`, `block_pose`, `goal_pose`, `n_contacts`, plus reward/terminated/truncated/step/id/render metadata.

Contact question: yes, there is enough state to compute contact-like geometry between the pusher and T-block. The `state` vector gives pusher xy, T-block xy, and T-block orientation. The default PushT variation randomizes start positions and block angle, not shape/scale, so the default geometry can be reconstructed from the environment code. The env also emits `n_contacts` directly in `info`; if that column is present in the official file, it is the cleanest contact label. If you need exact Pymunk contact manifolds rather than geometric contact/no-contact, replaying state in the env or using recorded `n_contacts` is safer than a hand-derived distance test.

## Model Architecture

Core code is in `jepa.py`, `module.py`, and `config/train/model/lewm.yaml`.

Architecture:

- `JEPA`: encoder + action encoder + autoregressive predictor + projector heads.
- Image encoder: `stable_pretraining.backbone.utils.vit_hf("tiny")`, patch size 14, image size 224, hidden size 192, 12 transformer layers, 3 attention heads, trained from scratch, `use_mask_token=false`.
- Image embedding: uses Hugging Face ViT `last_hidden_state[:, 0]` CLS token, then a 2-layer projector MLP `192 -> 2048 -> 192` with BatchNorm.
- Action encoder: `Embedder`; 1D kernel-1 conv from action-block dimension to `smoothed_dim=10`, then MLP to 192. PushT/TwoRooms/Reacher have input_dim 10; Cube has input_dim 25.
- Predictor: `ARPredictor`; learned positional embedding over history length 3; 6 conditional transformer blocks with hidden 192, 16 heads, dim_head 64, MLP dim 2048, dropout 0.1. Conditioning uses AdaLN-zero modulation from the action embedding.
- Prediction projection: another `192 -> 2048 -> 192` MLP with BatchNorm.
- Training loss: next-embedding MSE plus SIGReg Gaussian regularizer with weight 0.09.

## Minimal Inference Path

There are two checkpoint formats to keep straight:

1. Hugging Face LeWM mirrors ship `weights.pt` plus `config.json`.
2. `eval.py` and `stable_worldmodel` loading expect an object checkpoint named like `<run>_object.ckpt` under `$STABLEWM_HOME`.

For planning/evaluation after an object checkpoint exists:

```bash
python eval.py --config-name=pusht.yaml policy=pusht/lewm
```

The README shows a conversion path from HF `weights.pt` + `config.json` to `$STABLEWM_HOME/pusht/lewm_object.ckpt`. After conversion, `swm.policy.AutoCostModel("pusht/lewm")` or `swm.wm.utils.load_pretrained("pusht/lewm")` is the intended loader.

For direct frame encoding, the code path is:

1. Load/build the `JEPA` model.
2. Preprocess frames the same way as `eval.py`: `ToImage`, float scaling, ImageNet normalization from `stable_pretraining`, resize to 224.
3. Call `model.encode({"pixels": pixels})`, where `pixels` has shape `(B, T, 3, 224, 224)`.
4. Read `info["emb"]`, shape `(B, T, 192)`.

For predictor rollout:

1. Provide a 3-frame history to `model.rollout()`: `info["pixels"]` shaped `(B, S, history, 3, 224, 224)`.
2. Provide candidate action blocks shaped `(B, S, horizon, action_block_dim)`, where PushT `action_block_dim=10` because `frameskip=5` and raw action dim is 2.
3. `model.rollout()` encodes the initial history, repeatedly calls `action_encoder` + `predict`, and returns `info["predicted_emb"]`.

For a one-step predictor call without the planner wrapper:

- encode history with `model.encode`
- encode actions with `model.action_encoder`
- call `model.predict(emb[:, -3:], act_emb[:, -3:])`

## Dependencies and Non-CUDA / Apple Silicon Notes

The repo itself has no `pyproject.toml` or requirements file. README installation delegates to:

```bash
uv pip install stable-worldmodel[train,env]
```

Relevant dependencies from `stable-worldmodel` and `stable-pretraining` include:

- Core: `torch`, `torchvision`, `numpy`, `einops`, `pillow`, `gymnasium`, `lancedb>=0.30.0`, `pylance>=4.0.0`, `pyarrow`, `tqdm`, `typer`, `rich`.
- Training: `stable-pretraining>=0.1.7`, `transformers>=4.50.0`, `hydra-core`, `hydra-submitit-launcher`, `wandb`, `lightning`, `omegaconf`, `torchmetrics`, `scikit-learn`, `timm`.
- Envs: `pygame`, `pymunk`, `shapely`, `ogbench`, `minigrid`, `gymnasium[all]`, `gymnasium-robotics`, `opencv-python`, `stable_baselines3`, `craftax`, `ale-py`.
- Formats: `h5py`, `hdf5plugin`, `decord`, `imageio[ffmpeg]`.

CUDA assumptions / likely issues:

- `eval.py` hardcodes `os.environ["MUJOCO_GL"] = "egl"` and `model.to("cuda")`.
- `config/eval/solver/cem.yaml` and `adam.yaml` hardcode `device: "cuda"`.
- Training config uses Lightning `accelerator: gpu` and `precision: bf16`.
- The earlier hardcoded CUDA issue in `SIGReg` appears fixed: it now samples random projections on `proj.device`.

Apple Silicon / MPS implications:

- Out of the box, eval is not device agnostic. On a Mac/MPS or CPU-only system, at minimum the hardcoded `.to("cuda")` and solver `device: "cuda"` need overriding/patching.
- `MUJOCO_GL=egl` is Linux/NVIDIA-oriented; macOS MuJoCo/DMControl usually needs a different rendering path.
- PushT itself uses Pygame/Pymunk/OpenCV and is likely less CUDA-bound, but LeWM eval still routes through CUDA-specific config.
- PyTorch `scaled_dot_product_attention` is used in the predictor; modern PyTorch supports CPU/MPS paths, but performance and dtype support may differ from CUDA. BF16 training on MPS should not be assumed stable without testing.
